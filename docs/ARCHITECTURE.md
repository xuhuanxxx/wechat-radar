# Lark Radar 架构设计文档

> 版本: v1.0
> 日期: 2026-06-09
> 状态: 设计中

## 1. 设计目标

将 Lark Radar 从单体 Next.js 全栈应用解耦为三层架构：

1. **Web 层**: 无状态纯前端，可容器化部署
2. **数据层**: Go 编译的单二进制服务，内嵌于 macOS app
3. **UI 层**: Swift 菜单栏应用，负责管理数据层进程

## 3. 组件定义

### 3.1 Web 层 (Next.js)

| 属性 | 说明 |
|------|------|
| 技术栈 | Next.js 16 + React 19 + TypeScript |
| 状态 | 完全无状态，不访问文件系统 |
| 配置 | 仅 `DATA_API_URL` 环境变量 |
| 部署 | Docker 容器，任意主机 |
| 职责 | UI 渲染、用户交互、API 代理转发 |

**关键设计:**
- 所有 `/api/*` 请求通过 catch-all route 代理到数据层
- 不直接访问 SQLite、lark-cli、文件系统
- 构建产物为 standalone 模式，适合容器化

### 3.2 数据层 (Go)

| 属性 | 说明 |
|------|------|
| 技术栈 | Go 1.22+ + 标准库 net/http |
| 数据库 | SQLite (mattn/go-sqlite3) |
| 输出 | 单二进制文件，零运行时依赖 |
| 端口 | 默认 3456，可配置 |
| 职责 | 数据存储、飞书同步、情报生成、缓存 |

**关键设计:**
- 单二进制文件，启动时间 < 100ms
- 内存占用 < 50MB
- 支持并发请求（goroutine）
- CORS 允许所有来源（Web 容器跨域）

### 3.3 UI 层 (Swift)

| 属性 | 说明 |
|------|------|
| 技术栈 | Swift + Cocoa |
| 类型 | 菜单栏应用（Menu Bar Extra） |
| 职责 | 启动/停止数据层、状态监控、用户交互 |

**关键设计:**
- 启动时自动启动 Go 数据服务进程
- 崩溃自动重启（指数退避）
- 定时触发自动同步（根据配置）
- 菜单项：打开 Web、立即同步、设置、退出

## 4. 通信协议

### 4.1 Web → 数据层

- 协议: HTTP/1.1
- 格式: JSON
- 认证: 无（本地服务，信任网络）
- 跨域: CORS 允许所有来源

### 4.2 数据层 → lark-cli

- 协议: 本地进程调用（os/exec）
- 格式: JSON（--json 参数）
- 超时: 60s 默认

### 4.3 Swift → Go 进程

- 协议: NSTask / Process
- 通信: 标准输入/输出 + HTTP 健康检查
- 监控: 进程退出回调 + 定时心跳

## 5. 数据存储

### 5.1 SQLite 数据库

| 配置 | 值 | 说明 |
|------|-----|------|
| 路径 | `~/.lark-radar/radar.db` | 用户主目录 |
| 模式 | WAL | 写前日志，并发性能 |
| 同步 | NORMAL | 平衡性能与持久性 |
| 缓存 | 64MB | 负值表示 KB |
| mmap | 256MB | 内存映射 I/O |
| busy_timeout | 5000ms | 锁等待 |

### 5.2 配置文件

| 配置 | 路径 | 格式 |
|------|------|------|
| 主配置 | `~/.lark-radar/config.json` | JSON |
| 数据目录 | `~/.lark-radar/` | 目录 |

## 6. 安全设计

### 6.1 网络隔离

- 数据层默认监听 `127.0.0.1:3456`（仅本机）
- Docker 场景下需显式监听 `0.0.0.0:3456`
- Web 容器通过 `host.docker.internal` 访问

### 6.2 凭证管理

- lark-cli 认证信息存储在 macOS Keychain（由 lark-cli 管理）
- 不进入 Docker 容器
- 不暴露在环境变量中

### 6.3 数据隐私

- 所有数据本地存储（SQLite）
- 不上传云端
- 飞书 API 调用通过 lark-cli（用户已授权）

## 7. 容错设计

### 7.1 数据层崩溃恢复

```
Swift ProcessManager:
1. 检测到 Go 进程退出
2. 等待 5s（指数退避，最大 60s）
3. 重新启动 Go 进程
4. 重试 10 次后标记为失败，通知用户
```

### 7.2 Web 层不可用

- 数据层继续运行，同步不受影响
- 用户可通过菜单栏查看状态
- Web 容器可独立重启

### 7.3 数据库损坏

- WAL 模式降低损坏风险
- 启动时自动检查 `PRAGMA integrity_check`
- 损坏时尝试从 WAL 恢复，失败时提示用户

## 8. 扩展性设计

### 8.1 水平扩展（未来）

- 数据层可部署为远程服务（暴露公网 IP）
- Web 层可部署多个实例（无状态）
- 数据库可迁移到 PostgreSQL（Go 端抽象 DB 接口）

### 8.2 多平台支持

- Go 数据层可编译为 Windows/Linux/macOS
- Swift UI 层仅 macOS（未来可用 Tauri/Electron 替代）
- Web 层跨平台（浏览器即可）

## 10. 版本兼容性

### 10.1 API 版本控制

- URL 前缀: `/api/v1/...`（未来）
- 当前保持 `/api/...` 不变
- 响应包含 `api_version` 字段

### 10.2 数据库迁移

- 启动时自动执行迁移
- 版本号存储在 `meta` 表
- 不支持降级（备份恢复）

## 11. 决策记录

### ADR-001: 为什么选择 Go 而不是 Node.js

- **背景**: 数据层需要编译为单二进制，内嵌于 macOS app
- **选项**: Go、Rust、Python、Node.js (pkg)
- **决策**: 选择 Go
- **原因**:
  1. 编译为单二进制，无运行时依赖
  2. 标准库强大，HTTP/SQLite/并发原生支持
  3. 开发效率高于 Rust，性能足够
  4. 跨平台编译简单（GOOS/GOARCH）
- **后果**: 需要重写业务逻辑（约 25h）

### ADR-002: 为什么 Web 容器化

- **背景**: Web 层需要无状态，支持灵活部署
- **选项**: 内嵌于 macOS app、独立进程、Docker 容器
- **决策**: Docker 容器化
- **原因**:
  1. 真正无状态，可部署到任意环境
  2. 与数据层生命周期解耦
  3. 开发/测试/生产环境一致
  4. 未来可部署到服务器/云端
- **后果**: 需要 Docker 环境，增加一层网络跳转

### ADR-003: 为什么 Swift 菜单栏而不是 Tauri

- **背景**: 需要 macOS 原生体验，管理后台进程
- **选项**: Swift、Tauri、Electron、Go Systray
- **决策**: Swift + Cocoa
- **原因**:
  1. 原生 macOS 体验，资源占用最低
  2. NSTask 管理子进程成熟稳定
  3. 菜单栏 API 原生支持
- **后果**: 仅支持 macOS（可接受，数据层跨平台）
