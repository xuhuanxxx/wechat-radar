# Lark Radar 架构设计文档

> 版本: v2.0
> 日期: 2026-06-09
> 状态: 设计中

## 1. 设计目标

将 Lark Radar 解耦为两个独立组件：

1. **Web 服务**: 无状态纯前端，可部署到公网/内网/本地，通过 HTTP/WebSocket API 消费数据
2. **macOS 数据服务**: 仅运行在用户 Mac 上，负责 SQLite + lark-cli 同步，暴露 HTTP API

**关键原则**:
- macOS 数据服务不感知 Web 服务的存在、位置、入口
- Web 服务通过配置知道数据服务地址
- 两者完全独立部署，独立生命周期

## 3. 组件定义

### 3.1 Web 服务 (Next.js)

| 属性 | 说明 |
|------|------|
| 技术栈 | Next.js 16 + React 19 + TypeScript |
| 状态 | 完全无状态，不访问文件系统 |
| 配置 | `DATA_API_URL` 环境变量（指向用户 Mac） |
| 部署 | Docker 容器，任意主机（公网/内网/本地） |
| 职责 | UI 渲染、用户交互、API 代理转发 |

**关键设计:**
- 所有 `/api/*` 请求通过 catch-all route 代理到 `DATA_API_URL`
- 不直接访问 SQLite、lark-cli、文件系统
- 支持 HTTP 和 WebSocket API
- 构建产物为 standalone 模式，适合容器化

**入口灵活性:**
- 部署在公网：`https://lark-radar.example.com`
- 部署在公司内网：`http://lark-radar.company.local`
- 部署在本地电脑：`http://localhost:3000`
- 用户通过浏览器访问任意入口

### 3.2 macOS 数据服务 (Go)

| 属性 | 说明 |
|------|------|
| 技术栈 | Go 1.22+ + 标准库 net/http |
| 数据库 | SQLite (mattn/go-sqlite3) |
| 输出 | 单二进制文件，零运行时依赖 |
| 端口 | 默认 3456，可配置 |
| 部署 | 仅用户 Mac，通过 Swift 菜单栏管理 |
| 职责 | 数据存储、飞书同步、情报生成、缓存 |

**关键设计:**
- 单二进制文件，启动时间 < 100ms
- 内存占用 < 50MB
- 支持并发请求（goroutine）
- CORS 允许所有来源（Web 可能跨域）
- **不感知 Web 服务的存在**——只暴露 API，谁来调用不关心

### 3.3 Swift 菜单栏 (macOS)

| 属性 | 说明 |
|------|------|
| 技术栈 | Swift + Cocoa |
| 类型 | 菜单栏应用（Menu Bar Extra） |
| 职责 | 启动/停止数据服务、状态监控、用户交互 |

**关键设计:**
- 启动时自动启动 Go 数据服务进程
- 崩溃自动重启（指数退避）
- 定时触发自动同步（根据配置）
- **菜单项**: 立即同步、打开数据目录、设置、退出
- **没有"打开 Web"菜单项**——不感知 Web 存在

## 4. 通信协议

### 4.1 Web → 数据服务

- 协议: HTTP/1.1 或 WebSocket
- 格式: JSON
- 认证: 无（本地服务，信任网络）
- 跨域: CORS 允许所有来源
- 配置: Web 通过 `DATA_API_URL` 环境变量知道数据服务地址

```
Web 服务 (任意位置)          macOS 数据服务 (用户 Mac)
     │                              │
     │  GET /api/stats              │
     │─────────────────────────────→│
     │                              │
     │  { ok: true, ... }           │
     │←─────────────────────────────│
```

### 4.2 WebSocket API (实时推送)

```
Web 服务                          macOS 数据服务
     │                              │
     │  WS /ws/sync-progress        │
     │─────────────────────────────→│
     │                              │
     │  { type: "progress", ... }   │
     │←─────────────────────────────│
     │  { type: "finished", ... }   │
     │←─────────────────────────────│
```

**WebSocket 事件:**
- `sync-progress`: 同步进度推送
- `new-messages`: 新消息到达通知
- `mention`: @提及通知

### 4.3 数据服务 → lark-cli

- 协议: 本地进程调用（os/exec）
- 格式: JSON（--json 参数）
- 超时: 60s 默认

### 4.4 Swift → Go 进程

- 协议: NSTask / Process
- 通信: 标准输入/输出 + HTTP 健康检查
- 监控: 进程退出回调 + 定时心跳

## 6. 数据存储

### 6.1 SQLite 数据库

| 配置 | 值 | 说明 |
|------|-----|------|
| 路径 | `~/.lark-radar/radar.db` | 用户主目录 |
| 模式 | WAL | 写前日志，并发性能 |
| 同步 | NORMAL | 平衡性能与持久性 |
| 缓存 | 64MB | 负值表示 KB |
| mmap | 256MB | 内存映射 I/O |
| busy_timeout | 5000ms | 锁等待 |

### 6.2 配置文件

| 配置 | 路径 | 格式 |
|------|------|------|
| 主配置 | `~/.lark-radar/config.json` | JSON |
| 数据目录 | `~/.lark-radar/` | 目录 |

## 7. 安全设计

### 7.1 网络隔离

- 数据服务默认监听 `127.0.0.1:3456`（仅本机）
- 如需公网访问，用户需手动配置端口转发或 VPN
- Web 服务通过 `DATA_API_URL` 连接到数据服务

### 7.2 凭证管理

- lark-cli 认证信息存储在 macOS Keychain（由 lark-cli 管理）
- 不离开用户 Mac
- 不暴露在环境变量中

### 7.3 数据隐私

- 所有数据本地存储（SQLite）
- 不上传云端
- 飞书 API 调用通过 lark-cli（用户已授权）

## 8. 容错设计

### 8.1 数据服务崩溃恢复

```
Swift ProcessManager:
1. 检测到 Go 进程退出
2. 等待 5s（指数退避，最大 60s）
3. 重新启动 Go 进程
4. 重试 10 次后标记为失败，通知用户
```

### 8.2 Web 服务不可用

- 数据服务继续运行，同步不受影响
- 用户可通过菜单栏查看数据服务状态
- Web 服务可独立重启/重新部署

### 8.3 数据库损坏

- WAL 模式降低损坏风险
- 启动时自动检查 `PRAGMA integrity_check`
- 损坏时尝试从 WAL 恢复，失败时提示用户

## 9. 扩展性设计

### 9.1 多 Web 入口

- 同一数据服务可被多个 Web 服务连接
- 用户可在公司内网部署一个，公网部署一个
- 数据服务无状态感知，只处理 API 请求

### 9.2 多平台数据服务

- Go 数据服务可编译为 Windows/Linux/macOS
- Swift UI 层仅 macOS（未来可用其他方案替代）
- Web 服务跨平台（浏览器即可）

## 11. 决策记录

### ADR-001: 为什么选择 Go 而不是 Node.js

- **背景**: 数据服务需要编译为单二进制，内嵌于 macOS app
- **选项**: Go、Rust、Python、Node.js (pkg)
- **决策**: 选择 Go
- **原因**:
  1. 编译为单二进制，无运行时依赖
  2. 标准库强大，HTTP/SQLite/并发原生支持
  3. 开发效率高于 Rust，性能足够
  4. 跨平台编译简单（GOOS/GOARCH）
- **后果**: 需要重写业务逻辑（约 25h）

### ADR-002: 为什么 Web 与数据服务完全解耦

- **背景**: Web 需要灵活部署，数据服务必须本地运行
- **选项**: 内嵌、共享进程、完全独立
- **决策**: 完全独立，HTTP 通信
- **原因**:
  1. Web 可部署到公网/内网/本地，不受数据服务限制
  2. 数据服务只关心本地数据，不感知 Web 存在
  3. 用户可自由选择 Web 入口
  4. 安全边界清晰（数据不出本地）
- **后果**: 需要配置 `DATA_API_URL`，增加网络依赖

### ADR-003: 为什么 Swift 菜单栏不感知 Web

- **背景**: 菜单栏是数据服务的管理界面
- **选项**: 包含 Web 入口、不包含 Web 入口
- **决策**: 不包含 Web 入口
- **原因**:
  1. Web 入口由用户/管理员决定，可能变化
  2. 菜单栏只管理数据服务生命周期
  3. 避免硬编码 URL，保持灵活性
  4. 用户通过浏览器书签/收藏访问 Web
- **后果**: 用户需自行记住 Web 入口地址
